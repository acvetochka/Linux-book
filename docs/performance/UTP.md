# Методологія УМУН — докладна довідка

`УМУН` — це практична методологія аналізу та усунення проблем (troubleshooting), яка допомагає системно шукати причину несправності, перевіряти гіпотези та документувати результат.

Назва УМУН розшифровується як:

1. У — Уточнення проблеми
2. М — Моделювання причин
3. У — Усунення та перевірка
4. Н — Накопичення знань

Методологія часто використовується:

- у технічній підтримці,
- DevOps,
- системному адмініструванні,
- мережевій інженерії,
- підтримці Linux/Windows,
- роботі з БД,
- підтримці користувачів.

Головна ідея УМУН

Не можна:

- хаотично натискати кнопки,
- змінювати все підряд,
- перезапускати сервіси «на удачу».

Потрібно:

- зрозуміти проблему,
- побудувати гіпотези,
- перевіряти їх по одній,
- зафіксувати результат.

## 1. `У`— Уточнення проблеми
Мета етапу

Максимально точно зрозуміти:

- що саме не працює,
- де проблема,
- коли вона виникла,
- кого зачіпає,
- як її відтворити.

**Основні задачі етапу**
1. Зібрати симптоми

    Симптом ≠ причина.

    Наприклад:

    | Симптом               | Реальна причина            |
    | --------------------- | -------------------------- |
    | Сайт не відкривається | Nginx не запущений         |
    | База «висить»         | Закінчилось місце на диску |
    | SSH не працює         | Firewall блокує порт       |
    | MySQL не стартує      | Пошкоджений ibdata         |

2. Визначити масштаб проблеми

    Потрібно зрозуміти:

    - Один користувач чи всі?
    - Один сервер чи весь кластер?
    - Один сервіс чи вся система?
    - Проблема постійна чи періодична?

3. Визначити момент появи

    Дуже важливі питання:

    - Що змінилось?
    - Чи був deploy?
    - Чи оновлювали пакети?
    - Чи міняли конфіг?
    - Чи закінчився сертифікат?
    - Чи була аварія живлення?

4. Відтворення проблеми

    Якщо проблему можна стабільно відтворити —
    це вже половина рішення.

**На цьому етапі використовують**
- Linux
    ```bash
    systemctl status
    journalctl
    dmesg
    top
    htop
    ss
    netstat
    df -h
    free -m
    ```

- Мережа
    ```bash
    ping
    traceroute
    mtr
    tcpdump
    curl
    telnet
    nc
    ```

- Web
    ```bash
    curl -I
    curl -v
    ```

- Logs
    ```bash
    tail -f
    grep
    less
  ```

**Правильні питання користувачу**  
Погані

❌ «У вас щось не працює?»

Хороші

✅ «Яку саме помилку ви бачите?»

✅ «Після чого це почалось?»

✅ «Чи працювало раніше?»

✅ «Проблема у всіх користувачів?»

**Результат етапу Уточнення**

Має бути:

- чіткий опис проблеми,
- список симптомів,
- область проблеми,
- часові рамки,
- умови відтворення.

## 2. `М` — Моделювання причин
**Мета**

Побудувати гіпотези:

- що могло зламатися,
- де саме,
- чому це дає такі симптоми.

**Головне правило**

Не перевіряти хаотично.

Потрібно:

- скласти список можливих причин,
- відсортувати їх за ймовірністю,
- перевіряти по черзі.

**Приклад мислення**

Сайт не відкривається.

Можливі причини:

| Причина             | Ймовірність |
| ------------------- | ----------- |
| Nginx не працює     | Висока      |
| Backend недоступний | Висока      |
| DNS проблема        | Середня     |
| Firewall            | Середня     |
| SSL сертифікат      | Низька      |
| Проблема в браузері | Низька      |

**Принцип "від простого до складного"**

Спочатку перевіряють:

1. чи працює сервіс,
2. чи відкритий порт,
3. чи є мережа,
4. чи є ресурси,
5. чи правильний конфіг.

А вже потім:

- packet capture,
- debugging,
- strace,
- профілювання ядра.

**Типові категорії причин**
1. Конфігурація
   - неправильний nginx.conf
   - помилка в Apache
   - неправильний my.cnf
2. Мережа
    - firewall
    - DNS
    - routing
    - MTU
    - NAT
3. Ресурси
    - RAM
    - CPU
    - disk space
    - inode exhaustion
4. Права доступу
    - SELinux
    - chmod
    - chown
    - sudo
    - ACL
5. Сервіси
    - crashed process
    - deadlock
    - dependency failure

**Побудова дерева причин**

Наприклад:

Сайт не відкривається
```bash
│
├── DNS
├── Firewall
├── Nginx
│   ├── Config error
│   ├── Port busy
│   └── Service stopped
├── Backend
└── SSL
```

## 3. `У` — Усунення та перевірка
**Мета**

Не просто «щось виправити», а:

- усунути реальну причину,
- перевірити результат,
- переконатися, що проблема не повернеться.

**Правило**

Одна зміна → одна перевірка.

❌ Погано:
```bash
systemctl restart nginx
systemctl restart mysql
reboot
iptables -F
```

Після цього незрозуміло:

- що саме допомогло,
- чи проблема повернеться.

**Правильний підхід**
1. Внести зміну

    Наприклад:
    ```bash
    nginx -t
    ```
    Побачили помилку.

    Виправили конфіг.

2. Перевірити
```bash
systemctl reload nginx
curl -I localhost
```

3. Перевірити користувацький сценарій
```bash
curl https://site.com
```

4. Переконатися у відсутності побічних ефектів
   - CPU нормальний?
   - Logs чисті?
   - Інші сайти працюють?

**Важливість rollback**

Перед змінами:

- backup config,
- snapshot,
- git commit.

**Типові помилки**
1. Виправлення симптомів

    Наприклад:
    ```bash
    systemctl restart mysql
    ```
    але причина:
    - disk full.

2. Внесення багатьох змін

    Тоді неможливо знайти справжню причину.

3. Відсутність перевірки

    Проблема може повернутись через 5 хвилин.

## 4. `Н` — Накопичення знань
Найважливіший етап

Багато адміністраторів пропускають його.

Але саме тут:

- команда стає сильнішою,
- зменшується час простою,
- створюється база знань.

**Що потрібно документувати**
1. Симптоми
   - що бачив користувач,
   - які були помилки.
2. Причину

    Не «сайт лежав».

    А:

    > "Nginx не стартував через помилку ';' у конфігурації."

3. Як знайшли
      - які команди,
      - які логи,
      - які перевірки.
4. Як виправили
      - конкретні зміни,
      - команди,
      - rollback.
5. Як уникнути повторення

    Наприклад:

    - додати monitoring,
    - CI linting,
    - backup,
    - healthcheck.

## Приклад повного Troubleshooting по УМУН
**Ситуація**

Користувач каже:

"Сайт не відкривається."

### Етап 1 — Уточнення
Збираємо симптоми

Питання:

- У всіх чи тільки у вас?
- Яка помилка?
- Коли почалось?
- Що змінювали?

Виявлено
- Проблема у всіх.
- Помилка: 502 Bad Gateway.
- 20 хвилин тому був deploy backend.

Перевірка
```bash
curl -I https://site.com
```
Отримали:
> HTTP/1.1 502 Bad Gateway

Перевіряємо Nginx
```bash
systemctl status nginx
```
Nginx працює.

### Етап 2 — Моделювання

Можливі причини:

| Причина               | Ймовірність |
| --------------------- | ----------- |
| Backend впав          | Висока      |
| Неправильний upstream | Висока      |
| Firewall              | Низька      |
| SSL                   | Низька      |

Перевірка backend
```bash
ss -tulpn | grep 3000
```
Порт пустий.

Перевірка сервісу
```bash
systemctl status backend
```

Сервіс failed.

Logs
```bash
journalctl -u backend -n 50
```

Помилка:
> Database connection refused

Перевіряємо БД
```bash
systemctl status mysql
```
MySQL down.

Logs MySQL
```bash
journalctl -u mysql
```
Помилка:
> No space left on device

**Реальна причина**

Заповнився диск.

### Етап 3 — Усунення
Перевірка диска
```bash
df -h
```

> /dev/sda1 100%

Знаходимо великі файли
```bash
du -sh /var/log/*
```

Виявлено
> Log файл 40 GB.

Очищаємо
```bash
truncate -s 0 /var/log/mysql.log
```

Перевіряємо
```bash
df -h
```
Є місце.

Запускаємо MySQL
```bash
systemctl start mysql
```

Запускаємо backend
```bash
systemctl start backend
```

Перевірка
```bash
curl -I https://site.com
```

> HTTP/1.1 200 OK

### Етап 4 — Накопичення знань
**Документація інциденту**

Симптом
> 502 Bad Gateway.

Причина
> Disk full через log growth.

Виправлення
> Очищення логів + restart сервісів.

Профілактика
- logrotate
- monitoring disk usage
- alerting
- max log size

## 10-крокова модель траблшутінгу

це теж дуже поширений варіант методології траблшутінгу.
Фактично це більш практичне та операційне розширення УМУН, де кожен великий етап деталізується на окремі дії.

Якщо порівняти:

| УМУН              | 10-крокова модель                                      |
| ----------------- | ------------------------------------------------------ |
| Уточнення         | Підготовка, збір симптомів, відтворення                |
| Моделювання       | Звуження зони пошуку                                   |
| Усунення          | Виправлення, перевірка                                 |
| Накопичення знань | Запобігання повторенню                                 |
| —                 | План запобігання шкоди, техобслуговування, задоволення |


Така модель часто використовується:

- у enterprise support,
- NOC/SOC,
- Linux/Windows support,
- датацентрах,
- ISP,
- DevOps/SRE,
- техпідтримці hardware.

## Детальна методологія Troubleshooting у 10 кроків
### 1. Підготовка
**Мета**

Перед початком роботи потрібно:

- зрозуміти критичність проблеми,
- отримати доступи,
- підготувати інструменти,
- оцінити ризики.

**Що входить у підготовку**
1. Визначення пріоритету  
    Наприклад:

    | Ситуація                    | Пріоритет |
    | --------------------------- | --------- |
    | Не працює production        | Critical  |
    | Проблема одного користувача | Low       |
    | Впав кластер БД             | Emergency |

2. Отримання доступів

    Linux:
    ```bash
    ssh admin@server
    sudo -i
    ```

3. Підготовка інструментів 
   
    Linux troubleshooting toolkit
    ```bash
    top
    htop
    iotop
    vmstat
    iostat
    tcpdump
    curl
    ss
    journalctl
    dmesg
    ```

4. Вивчення архітектури

    Перед аналізом треба зрозуміти:
    ```
    User
    ↓
    Load Balancer
    ↓
    Nginx
    ↓
    Backend
    ↓
    MySQL
    ```
    Інакше troubleshooting буде хаотичним.

**Типова помилка**

❌ Одразу щось перезапускати.

### 2. План запобігання більшої шкоди
**Мета**

Не допустити:

- втрати даних,
- cascade failure,
- падіння інших систем.

**Приклади**

База даних переповнена

НЕ можна:
```bash
rm -rf /var/lib/mysql/*
```

RAID деградував

НЕ можна:

- перезавантажувати сервер,
- запускати fsck без аналізу.

**Дії захисту**
1. Backup
    ```bash
    cp nginx.conf nginx.conf.bak
    ```

2. Snapshot

    VM snapshot / LVM snapshot.

3. Виведення ноди з балансування

    Наприклад:
    > Disable node in HAProxy

4. Read-only режим

    Для БД.

**Головний принцип**

Спочатку стабілізувати ситуацію —
потім ремонтувати.

### 3. Збір інформації про симптоми
**Мета**

Отримати максимально повну картину.

**Джерела інформації**
1. Користувачі  
    Питання
    - Яка помилка?
    - Коли почалось?
    - Постійно чи інколи?
    - Що робили перед цим?
2. Monitoring
      - Zabbix
      - Prometheus
      - Grafana
      - Datadog
3. Logs
    ```bash
    journalctl
    tail -f
    grep ERROR
    ```
4. System metrics
    ```bash
    top
    free -m
    df -h
    ```
**Важливо**

На цьому етапі:  
❌ не виправляють,  
❌ не перезапускають,  
❌ не змінюють конфіг.  

### 4. Відтворення симптому
**Мета**

Побачити проблему власноруч.

**Приклади**

- Web
    ```bash
    curl -v https://site.com
    ```

- DB
    ```bash
    mysql -u root -p
    ```

- Network
    ```bash
    ping
    mtr
    tcpdump
    ```

**Чому це критично**

Якщо проблема:

- не відтворюється,
- нестабільна,
- випадкова,

то troubleshooting ускладнюється.

**Важливе правило**

Потрібно:

- знати як викликати проблему,
- знати як зрозуміти що вона зникла.

### 5. Виконання технічного обслуговування
**Мета**

Усунути очевидні базові проблеми.

**Типові дії**  
Disk cleanup
```bash
journalctl --vacuum-time=7d
```

Restart stuck process
```bash
systemctl restart nginx
```

Оновлення кешів
```bash
systemd-resolve --flush-caches
```

Rotation logs
```bash
logrotate
```

**Чому це окремий етап**  

Іноді проблема:

- не в архітектурі,
- не в багу,
- а у відсутності базового maintenance.

**Але є небезпека**

❌ Не можна перетворювати maintenance на:

«перезавантажимо все підряд».

### 6. Звуження зони пошуку
Найважливіший етап

Тут відбувається справжній troubleshooting.

**Мета**

Визначити:

- де саме проблема,
- який компонент винен.

**Метод поділу системи**

Приклад
> Сайт не працює.

Перевіряємо:
```
DNS?
↓
Load Balancer?
↓
Nginx?
↓
Backend?
↓
Database?
```

**Приклад перевірок**  
DNS
```bash
dig site.com
```

Port
```bash
ss -tulpn
```

Backend
```bash
curl localhost:3000
```

DB
```bash
mysqladmin ping
```

**Метод половинного поділу**

Дуже важливий принцип.

Не перевіряти все підряд.

А:

- ділити систему навпіл,
- локалізувати область проблеми.

### 7. Виправлення несправності
**Мета**

Усунути root cause.

**Важливо**

Root cause ≠ symptom.

**Приклад**

Симптом
> 502 Bad Gateway 

Root cause
> MySQL down через full disk

**Правильний процес**
1. Backup
2. Внести одну зміну
3. Перевірити

**Типові помилки**

❌ Внести 20 змін одразу.

❌ Видалити логи без аналізу.

❌ Reboot without investigation.

### 8. Перевірка
**Мета**

Довести, що:

- проблема вирішена,
- побічних ефектів нема.

**Рівні перевірки**
1. Технічна
    ```bash
    systemctl status
    curl
    ss
    ```
2. Функціональна

    Чи працює бізнес-функція?

3. Навантажувальна

    Проблема не повертається під load?

4. Моніторинг

    Графіки стабільні?

**Типова помилка**

❌ Перевірити лише:
```bash
service started
```

Але користувачі все ще мають помилки.

### 9. Отримання задоволення

Це напівжартівливий, але важливий етап.

Сенс етапу
- зафіксувати успіх,
- повідомити команду,
- закрити інцидент,
- знизити стрес.

**У production це важливо**

Troubleshooting:

- виснажує,
- створює panic mode,
- викликає помилки через стрес.

Тому після вирішення:

✅ підтверджують відновлення сервісу  
✅ повідомляють stakeholders  
✅ оновлюють incident status  

### 10. Запобігання повторенню проблеми
Найцінніший етап

Саме тут команда росте.

**Що потрібно зробити**
1. Root Cause Analysis (RCA)

    Не:
    > "Сервіс впав"

    А:
    > "Disk full через відсутність logrotate."

2. Документація
      - symptoms
      - diagnostics
      - commands
      - solution

3. Автоматизація  
    Наприклад

    Було:
    ```bash
    df -h
    ```
    Стало:
    - alert at 80%.

4. Preventive measures
